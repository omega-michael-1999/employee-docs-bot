EI-5 new_employee_correct Callback Data Is Never Handled

HIGH SEVERITY Engineering HIGH CONFIDENCE

Problem: When a user clicks "👎 No" during the confirmation flow, the bot asks "What's the employee's full name?" with a "🆕 New Employee" button whose callback_data is "new_employee_correct" (bot.py:762-766). When the user clicks this button, handle_callback is invoked with data = "new_employee_correct". The handler only checks for data == "new_employee" (bot.py:737) — but the sent callback data is "new_employee_correct". The check fails, and the code falls through to the else branch:

await query.edit_message_text("Sorry, this expired. Please send the document again.")

The session is still alive in chat_states — the user sees "expired" despite having a perfectly valid session. This is a user-visible bug: the "New Employee" option during the correction flow silently breaks.

This was introduced during the state machine refactor. In the original v1 code, handle_callback checked for "new_employee_correct" in the generic pending-dict scan path — but the new per-chat state machine changed the dispatching logic and the second callback data value was not carried over.

Proposed Solution

Add handler for new_employee_correct. In handle_callback, after the existing data == "new_employee" check (line 737), add an elif data == "new_employee_correct" branch that sets the session's awaiting = "employee_name" and sends the "What's the new employee's full name?" prompt — identical logic to the data == "new_employee" case at lines 738-745. One additional branch, ~8 lines.

Designer's response: Accepted.

Designer's response: Accepted.

QF-3 Discrepant Claude Haiku Model Versions

LOW SEVERITY Operational MEDIUM CONFIDENCE

Problem: bot.py (LLM classification tier) specifies model claude-3-haiku-20240307, while scripts/doc-extract.py (vision OCR) specifies claude-haiku-4-5-20251001. These are different model versions — the OCR script uses a newer Haiku than the classification tier. This may be intentional (older Haiku for text, newer for vision) but it's undocumented, so a future maintainer won't know whether to update one, both, or neither.

Both are valid API model IDs. This doesn't cause a runtime error, but the discrepancy should be resolved or documented. If the newer model works better for text classification (likely, since classification is simpler than vision), the bot should be using the same version. If the older model is deliberately pinned for cost or consistency, that should be noted in a comment.

Proposed Solution

Either: (a) unify on the newer model (claude-haiku-4-5-20251001) since it's cheaper and more capable, or (b) add a one-line comment next to the model string in classify_by_llm explaining why it's pinned to an older version: # Pinned to 20240307 for deterministic classification; newer models change output format. 30 seconds of documentation that saves a future developer from wondering.

Designer's response: Use the latest version in both places. Solution selected: option (a) — unify on claude-haiku-4-5-20251001 across both bot.py and doc-extract.py.

OP-3 No External Monitoring Despite Multi-API Dependencies

LOW SEVERITY Operational MEDIUM CONFIDENCE

Problem: The bot now depends on four external APIs (Telegram, Anthropic, Google Drive, and Haiku OCR). If any goes down, the bot silently degrades or fails. systemd's Restart=always handles process crashes, but it can't detect an API key expiring, a quota being exhausted, or the bot entering a crash-restart loop. The Telemetry section in architecture.html still lists "Service health: systemd auto-restart (RestartSec=10). No external monitoring."

For a compliance-critical document processing system, a silent outage that lasts days (e.g., Anthropic key expires while the admin is on vacation) means documents pile up unfiled. The DSHS compliance risk the bot was designed to mitigate reappears in a different form: not "documents not filed properly" but "documents not filed at all."

Proposed Solution

Minimum: heartbeat check. Add a simple health check endpoint or use a cron job that sends a daily test message to the Omega admin's Telegram if the bot has processed 0 documents in 24 hours. Even a journalctl --since="24 hours ago" | grep "Document processed" | wc -l cron check that emails zero-counts is better than nothing.

Better: uptimerobot.com or similar. The bot's polling connection to Telegram doubles as a health signal — if the bot is alive, Telegram API responds. A free external monitoring service that pings the server every 5 minutes and alerts via SMS/email when journald stops producing log lines would catch crashes, hangs, and restart loops.

This is low severity because the bot hasn't launched yet. But it should be in the pre-production checklist.

Designer's response: Set up UptimeRobot monitoring for the system and dependencies. User will provide an API key for setup.

QF-4 No E2E Test for the Full Pipeline

MEDIUM SEVERITY Operational MEDIUM CONFIDENCE

Problem: The v1 PRD listed "End-to-end: Send a real document via test Telegram channel, verify classification, confirmation, and Drive filing" as the highest-priority test seam. The new unit tests (test_state_machine.py, test_roster_cache.py, test_llm_parse.py) are well-structured and valuable, but there is still no end-to-end test that exercises the full pipeline: Telegram message → download → text extraction → classification → confirmation → Drive filing. The handle_document and handle_callback functions — which contain the orchestration logic — have no tests at all.

The callback bug (EI-5) would have been caught by even a minimal E2E test that sends a mock callback with new_employee_correct data and asserts a valid response. The classification tier selection logic in classify() has tests for the rules tier but the orchestration between rules → LLM → manual in the actual handler is untested.

Proposed Solution

Add integration tests for the handler orchestration layer. Using python-telegram-bot's test utilities (MockTelegramBot, Update helpers), write tests that:

  • Send a mock document update and assert the correct state transition and confirmation message
  • Send a mock callback with confirm_yes and verify the session enters FILING state
  • Send a mock callback with confirm_no and verify the correction flow is initiated
  • Send a mock callback with new_employee_correct and assert it doesn't return "expired"
  • Send text with no pending state and assert silent ignore

Leave Drive and API integration to manual end-to-end testing (those are hard to mock usefully), but the Telegram handler orchestration is pure state machine — testable without any external dependency.

Designer's response: Accepted. Created a bot called edmonds_villa_testing_bot that is a member of both employee-docs channels for posting test messages. E2E tests should cover all workflows, variations, and edge cases, and must be run against the afh-22 instance — not the Edmonds Villa production instance.

The code quality is visibly higher: explicit state machine, persistent storage, cache with TTL, doc-level observability, proper temp file management, working tests from any checkout path.
