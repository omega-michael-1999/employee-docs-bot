Good Foundation — Moderate Issues to Resolve Before Production
The document processing pipeline is well-conceived, the multi-tenant config is clean, and the hybrid classification tier logic makes economic sense. Two structural problems — state loss on restart and single service account isolation — must be resolved before this bot serves real AFH compliance data. The engineering-level issues (fragile state machine, cache absence, race conditions) will compound rapidly under real usage. Fix these and this is a solid B+/A- design.

Key takeaway: Don't put this into production with in-memory state and a shared service account. Those two choices turn every crash into a data-loss event and every tenant into a cross-tenant liability.

What Works

✅ Clean Multi-Tenant Configuration

The config.json client array pattern is exactly right. Adding a new AFH client means adding one JSON block — no code changes, no redeployment. The separation of per-instance .env files from shared config is the correct boundary. This is the strongest architectural decision in the design.

✅ Three-Tier Classification Is Cost-Effective

Rules → LLM → manual is the right economic structure. Rules cost nothing and handle the common case. LLM costs pennies only for ambiguous documents. Manual correction is free for the rare failures. The PRD correctly identifies that most documents will be routine (CPR renewal for a known employee) and will hit Tier 1 with high confidence. The Tier 2 validator that checks LLM output against known rosters is a critical safety net.

✅ Document Extraction Subprocess Isolation

Running doc-extract.py as a subprocess with a 300-second timeout is the right call. Document extraction is I/O-heavy, memory-variable (a 50-page scanned PDF), and failure-prone (corrupted files). Isolating it from the main bot process prevents a hung extraction from blocking Telegram message handling. The fallback chain (PyPDF2 → PyMuPDF render → Haiku vision) is thorough.

✅ Companion -robot.txt for Searchability

Writing extracted text alongside every document is a simple insight with outsized value. Drive's native search doesn't index image text. The companion file makes every document discoverable by content. This alone justifies the OCR cost.

✅ HEIC Support at the Right Layer

iPhone users (overwhelmingly likely for AFH administrators) send HEIC by default. Converting in doc-extract.py via pillow-heif before the vision API call keeps the pipeline clean. The graceful degradation when the library is missing is a nice touch.

✅ Production-Grade Deployment Model

systemd with Restart=always, passwordless sudo for specific commands, per-token systemd instances, journald logging — this is how you deploy a single-server bot. The setup-clients.sh script for multi-client systemd management shows foresight.

Findings

SP-1 Roster Cache Absence Creates Latency-Scaling Problem

HIGH SEVERITY Structural HIGH CONFIDENCE

Problem: The employee roster is scanned from Google Drive for every single document. Each scan requires two Drive API calls: (1) find the CAREGIVERS folder, then (2) list its subfolders. For a low-volume AFH (5-10 documents/day) this is an acceptable latency tax of 1-2 seconds per document. But the design explicitly makes this a per-document cost with no caching, justified as "always fresh" (see architecture.html, State Management table).

The roster changes only when the Omega administrator manually adds a new employee folder to Drive — a rare, human-initiated event that happens at most a few times per month. Between those events, the roster is identical for every document processed. The "always fresh" argument collapses under this reality: the source of truth is mutated by human action, not by a concurrent process. The cache invalidation problem here is trivial — you don't need eventual consistency for a folder list that changes monthly.

At multi-tenant scale (3-5 AFHs, 50+ documents/day), this multiplies into measurable Drive API overhead and a real latency cost for the user waiting on a Telegram confirmation prompt. The Drive API quota (default 1B requests/day) won't be hit, but the per-document latency is additive and unnecessary.

Proposed Solution

In-memory cache with 30-minute TTL. Cache the employee roster dict on first fetch. On cache miss (TTL expiry or cold start), auto-re-fetch from Drive automatically — no manual reload command needed. This costs ~10 lines of code, eliminates the Drive API call from the hot path of every document, and the "staleness window" is bounded by a TTL the operator controls.

Designer's response: Approved — 30-minute TTL cache with automatic reload on cache miss. The administrator should not need to know about caching. Solution revised per designer: removed manual /reload-roster command, system re-fetches automatically on cache miss.

SP-2 State Loss on Restart Is a v0 Tradeoff Masquerading as v1

HIGH SEVERITY Structural HIGH CONFIDENCE

This is the most dangerous hidden problem in the design.

The PRD accepts in-memory state as a v1 decision: "restart = re-send document" (see PRD, Out of Scope). But this bot runs under systemd with Restart=always and RestartSec=10. Any crash — and this bot will crash, because it calls three external APIs (Telegram, DeepSeek, Anthropic, Google Drive) plus a subprocess for document extraction — results in the pending dict being obliterated. Within 10 seconds the bot is back up with zero knowledge of the document being processed.

Problem: The failure chain is: AFH administrator sends a photo of a CPR card → bot downloads, extracts text, classifies, and shows "Is this a CPR card for Fatou Manneh?" with [Yes][No] buttons → 5 seconds later, the bot crashes because DeepSeek's API returned a 502 or the temp file cleanup threw an exception → systemd restarts the process → the pending dict is empty → the user taps "Yes" → the bot responds "Sorry, this expired. Please send the document again." — the user has to re-photograph a physical paper document that's already in their pocket.

This isn't an edge case. This is the steady state for any long-running Telegram bot that integrates with four external APIs, each with its own failure modes. The PRD frames state loss as an acceptable "restart = re-send" tradeoff, but that framing assumes a clean restart (operator-initiated), not a crash restart (the default operational mode). For a compliance-critical document workflow where the user is physically handling paper records, forcing a re-send on every crash is a product failure.

Proposed Solution

SQLite-based persistence. Replace the in-memory pending dict with a simple SQLite database (stdlib, zero dependencies). The schema is trivial: chat_id, message_id, file_path, file_name, text, employee, category, state, created_at. A single pending.db file survives crashes and restarts. The state machine transitions become SQL UPDATEs. On bot startup, any pending documents older than 24 hours are cleaned up with a log warning. This costs ~40 lines of code and eliminates the data-loss-on-crash vulnerability entirely. The SQLite file writes to /opt/employee-docs-bot/data/, which is persistent.

Alternative (lighter): JSON file persistence. Serialize pending to pending.json on every state transition. On startup, deserialize if the file exists. Simpler but loses SQL queryability — and you will want to query pending documents when debugging.

Designer's response: Approved — go with SQLite persistence solution.

SP-3 Single Service Account Breaks Multi-Tenant Security Isolation

HIGH SEVERITY Structural HIGH CONFIDENCE

Problem: The architecture uses one Google service account shared by all clients, with "Editor" access on all client Drive folders and folder-level permissions as the only isolation boundary (see architecture.html, Security Model). This means any of the following events compromise every tenant's data:

  • The service account key file is leaked (stolen from the server, committed to git by accident, exposed via a debug endpoint)
  • A bug in the bot's Drive routing logic accidentally lists or writes to the wrong client's Drive folder
  • A malicious actor who gains access to one client's Telegram channel can use the bot as a relay to access other clients' Drive data
  • The service account has Editor access across all folders — a misrouted delete operation cannot be undone

The config schema's service_account_key_file field is per-client, which implies the designer considered per-client keys but didn't implement them. The architecture document and the file-naming convention (.service-account-keys/ directory) suggest a single shared key. This ambiguity is itself a finding — the config schema and the implementation disagree on the security model.

For a system handling compliance documents — government-issued IDs, background checks with fingerprints, TB test results, employment eligibility forms — this is a compliance risk under any reasonable interpretation of HIPAA and Washington State privacy regulations. The DSHS violation risk that the bot is designed to reduce (WAC 388-76-10198) is replaced by a data breach risk that's introduced by the shared-key architecture.

Proposed Solution

Per-client service accounts. Create one Google service account per AFH client, grant it Editor access only to that client's Drive root folder, and store the key file per-client as the config schema already implies ("service_account_key_file": ".service-account-keys/edmonds-villa-sa.json"). The code already supports this — it reads the key file from the per-client config entry. The only change is operational: create separate keys and grant scoped permissions. This eliminates cross-tenant blast radius. Key files remain gitignored and filesystem-protected.

Designer's response: Accepted.

SP-4 Handler State Machine Is Fragile and Race-Prone

HIGH SEVERITY Structural HIGH CONFIDENCE

Problem: The handle_callback and handle_text functions implement a state machine using ad-hoc dictionary lookups over the pending dict. The state encoding is inconsistent — sometimes keyed by (chat_id, message_id) (normal confirmations), sometimes by (chat_id, "awaiting_employee") (correction flows), and sometimes by (chat_id, "awaiting_new_name"). The handler tries pending.pop(key), then checks a specific string key, then iterates through the entire dict looking for any matching chat_id. This is a state machine implemented as a sequence of fragile lookups.

The specific race condition: handle_text iterates pending.keys() and grabs "the first pending key matching the chat_id" (bot.py:441-444):

for k in list(pending.keys()):
    if k[0] == chat_id:
        info = pending.pop(k)
        break

If a user has two documents in flight and types text, the handler grabs whichever pending entry comes first in dict iteration order (Python 3.7+ insertion order, meaning the oldest document). The text correction applies to the wrong document. This is user-visible data corruption.

Python's GIL protects individual dict operations but not compound check-then-pop sequences. With python-telegram-bot's async architecture handling callbacks and text messages concurrently, two handlers can operate on the same chat_id's state simultaneously — leading to documents being filed under wrong names, duplicate filings, or lost states.

Proposed Solution

Per-chat state machine with explicit state enum. Restructure to use a single state object per chat_id:

class ChatState(Enum):
    IDLE = auto()
    AWAITING_CONFIRMATION = auto()
    AWAITING_EMPLOYEE_CORRECTION = auto()
    AWAITING_DOCUMENT_TYPE = auto()
    FILING = auto()

chat_states: dict[int, ChatSession] = {}  # chat_id -> ChatSession

This eliminates the multi-key lookup pattern, ensures one state per chat (preventing cross-document confusion), and makes state transitions explicit and auditable. A per-chat asyncio.Lock prevents concurrent state mutations on the same chat_id. The lock is the cheap, correct fix for the race condition — it serializes state transitions for a given chat without blocking other chats.

Designer's response: Accepted.

EI-1 DeepSeek Classification Adds API Surface Without Clear Advantage

MEDIUM SEVERITY Engineering MEDIUM CONFIDENCE

Problem: The Tier 2 classification uses DeepSeek Chat API, but you already have Anthropic API keys for vision OCR. Adding DeepSeek creates a second external API integration with its own failure modes (different auth, different rate limits, different API versioning, different outage patterns) for a task any capable model could handle. The classification prompt is simple — extract employee name, category, and description from document text — well within the capability of Claude Haiku (which you're already paying for per-document for vision OCR).

The DeepSeek JSON response parsing is also fragile. The regex r'\{[^}]+\}' (bot.py:216) will fail on any JSON with nested objects or arrays. The model is instructed to return flat JSON, but this is enforced by prompt only, not by any structured output mechanism. If DeepSeek returns {"employee": "Fatou Manneh", "category": "04 - CPR & First Aid", "description": "CPR card, renewed 2026"}, it works. If it adds an array or wraps it in a markdown code block, it silently fails and degrades to Tier 3.

Proposed Solution

Use Claude Haiku for both OCR and classification. Since you're already paying for one Haiku call per scanned document, adding a second Haiku call (using the same API key, same client) for classification when the rules tier is ambiguous simplifies the architecture: one fewer API key to manage, one fewer provider to monitor, one fewer error-handling path. If Haiku costs are a concern (unlikely at the volume of ambiguous documents), cache the classification result — the same document text will always classify the same way.

If keeping DeepSeek: Replace the regex JSON parser with a proper json.loads(content) with cleanup for markdown code fences. Add a retry on JSON parse failure with a prompt correction. Add structured output via a function-calling API if DeepSeek supports it.

Designer's response: Accepted.

EI-2 Hardcoded Deployment Path Blocks Local Development

MEDIUM SEVERITY Engineering HIGH CONFIDENCE

Problem: Every test file starts with sys.path.insert(0, '/opt/employee-docs-bot'), meaning tests can only run on the deployment server at that exact path or with a symlink hack. The repo lives at subrepos/employee-docs-bot/ in this workspace. Running pytest locally fails immediately with an import error. This creates a development anti-pattern where tests are never run locally and code is only tested on the production server — or, more likely, never tested at all.

The classification tests (test_classification.py, test_ambiguity.py) are actually well-structured and valuable. They test the word-boundary matching logic, category keyword matching, and confidence scoring — all critical to the rules tier's accuracy. But they're effectively disabled by the path assumption.

Proposed Solution

Relative import with pytest path config. Either: (a) add a pytest.ini or pyproject.toml with pythonpath = ["."], or (b) restructure bot.py into a proper package (employee_docs_bot/__init__.py) with importable modules. The pytest config approach is lighter and sufficient for a project this size:

# pytest.ini (in repo root)
[pytest]
pythonpath = .

Change the test imports to relative: from bot import classify_by_rules works when pythonpath = . is set. Remove the sys.path.insert lines from all test files.

Designer's response: Accepted.

EI-3 Broad Message Filters Capture Unintended Messages

MEDIUM SEVERITY Engineering HIGH CONFIDENCE

Problem: Two filter issues in bot.py:667-669:

1. Text handler swallows all non-command messages. The correction text handler uses filters.TEXT & ~filters.COMMAND — which means any text message sent to a configured chat gets consumed by the correction flow, even if the user is just chatting. In a multi-tenant AFH channel where the administrator might send "I'll be in at 10 tomorrow" or "Did you get the Smith file?", that message vanishes into the state machine with no feedback. The handler silently pops from pending and, finding nothing, returns silently (if not info: return at bot.py:446). The user never sees their message again — it's been consumed by the bot with no response.

2. No file size limits or type restrictions. The document handler accepts filters.PHOTO | filters.Document.MimeType("application/pdf") | filters.Document.Category("image/"). A 100MB video mislabeled as image/png, a corrupted PDF that crashes doc-extract.py, or a HEIC photo at 48MP resolution all pass through to the processing pipeline with no size guard. They consume temp space, CPU for extraction, and API credits for the vision call, then fail with a generic error.

Proposed Solution

1. Scoped text handler: Only enter the correction handler when there's a pending state for that chat_id. Check chat_id in chat_states or chat_id in pending at the top of handle_text. If no pending state exists, ignore the message silently (or reply "I don't understand text messages outside of document correction flow").

2. File size and type guards: Reject files >20MB before download. Check the file extension against a whitelist of supported types before passing to doc-extract.py. Give clear error messages: "This file type is not supported. Please send PDF, JPEG, PNG, or HEIC photos of documents." instead of "Could not extract text from this document."

Designer's response: Both solutions accepted.

EI-4 No Document-Level Observability for Flow Tracing

MEDIUM SEVERITY Engineering HIGH CONFIDENCE

Problem: Logging uses plain Python logging.INFO with unstructured string messages. There's no document ID or correlation ID flowing through the pipeline. When a classification fails, the log shows Classification failed — manual correction needed (bot.py:259) but doesn't include the document name, chat context, or extracted text excerpt. When the DeepSeek API times out, the log shows DeepSeek classification failed: {e} but not what document was being classified. Debugging a production issue requires correlating journald timestamps with Telegram messages manually.

The architecture doc's Observability section (architecture.html, §Observability) lists logging, failed classification WARNINGs, and Drive error reporting — all of which are accurate, but none of which include a document ID or structured context.

Proposed Solution

Add a document UUID at pipeline entry. Generate a UUID (doc_xxx) when handle_document receives a file. Pass it through every log line as doc_id. Every state in the pipeline — received, extracted, classified (with tier and result), confirmed, filed — is logged with the doc_id. This makes journalctl -u employee-docs-bot | grep doc_abc123 the debugging superpower. The UUID also serves as the temp file name prefix, making it easy to correlate filesystem artifacts with log entries. This costs ~5 lines of code for the UUID generation and a logging.LoggerAdapter wrapper.

Designer's response: Accepted.

OP-1 Temp File Cleanup Is Non-Atomic and Leak-Prone

LOW SEVERITY Operational HIGH CONFIDENCE

Problem: Temp files are created with tempfile.NamedTemporaryFile(delete=False) (bot.py:298) to survive across async boundaries. Cleanup happens in try/finally blocks across handle_document and file_document, but the error handling is inconsistent:

  • In handle_document, temp files are cleaned up in file_document's finally block
  • But handle_document also has its own cleanup paths on early error returns (lines 308, 316, 325)
  • If the user never responds to a confirmation prompt, the temp file lives in /tmp forever
  • If the bot crashes between download and filing, the temp file is orphaned
  • The file_document function also creates a second temp file (-robot.txt) which it cleans up, but if the robot.txt upload succeeds and the main file upload fails, the robot.txt cleanup still runs, leaving the main file

In practice, /tmp is cleaned on reboot, and document volumes are low enough that leak accumulation is slow. But at production scale (hundreds of documents), orphaned temp files become a real disk pressure issue, especially for multi-page PDFs and high-resolution photos.

Proposed Solution

Register temp files in a cleanup registry. Track every created temp file path in a set(). On bot startup (or in a periodic async task), sweep and remove any files older than 24 hours. Use @contextmanager or a dedicated TempFileManager class that ensures cleanup in all exit paths. This is low-priority but eliminates an entire class of operational debt.

Designer's response: Accepted.

QF-1 User Gets No Feedback When Tier 2 (LLM) Falls Back to Tier 3

MEDIUM SEVERITY Operational MEDIUM CONFIDENCE

Problem: When DeepSeek fails (timeout, parse error, API down), the classification silently degrades to Tier 3 (manual correction). The user sees the same flow as if the bot never attempted automated classification — they get dumped into "What's the employee's full name?" with no context about why. The difference in user experience between "tap Yes/No" and "type full name, then type document type" is enormous. The user has no way to distinguish "the bot couldn't figure it out from the document text" (expected for low-quality photos) from "the LLM API is down" (transient, retry might work).

There's also a timing issue: the DeepSeek API call has a 30-second timeout. The user sees "Processing document..." and then waits up to 30 seconds with no intermediate feedback before getting dumped into the manual correction flow. Without explanation, this feels like the bot hung.

Proposed Solution

When Tier 2 fails and Tier 3 initiates, send an intermediate message: "I couldn't auto-classify this document (the classification service is temporarily unavailable). Let me ask you instead:" before starting the manual prompt. If the timeout is hit, tell the user "The classification service timed out — sorry about that. Let me ask you directly:" rather than leaving them waiting. This costs a couple of log checks and a reply_text call but dramatically changes the user's perception of reliability. A retry button for the LLM classification would be even better — let the user decide whether to wait for an API retry or skip to manual.

Designer's response: Accepted.

QF-2 What File Types Actually Reach the Pipeline?

LOW SEVERITY Operational LOW CONFIDENCE

Problem: The Telegram handler filter is filters.PHOTO | filters.Document.MimeType("application/pdf") | filters.Document.Category("image/"). But doc-extract.py only handles specific extensions: jpg, jpeg, png, tiff, tif, bmp, gif, webp, heic, heif, pdf, docx, txt, html, htm, xml, json, csv. The intersection of "MIME types Telegram says are images" and "file types doc-extract.py can handle" is not explicitly defined. If Telegram starts reporting a new image MIME type (e.g., image/avif for AVIF format), the document passes the filter but doc-extract.py's extension-based routing won't parse it — it falls through to the else block at line 62 which tries all methods, eventually returning None, and the user gets "Could not extract text from this document."

Proposed Solution

Define a single explicit whitelist of supported extensions (e.g., SUPPORTED_EXTS = {'.pdf', '.jpg', '.jpeg', '.png', '.heic', '.heif', '.docx', '.txt'}) and validate at the handler level. Reject unsupported types with a clear message: "I can only process PDF, JPEG, PNG, HEIC photos, DOCX, and TXT files." The error message should tell the user what they can send, not just what they sent wrong. This aligns with EI-3's file guard suggestion.

Designer's response: Accepted. The error message should say what type of file the system thinks the user provided, along with the list of file types the system can accept. Solution augmented per designer: include detected file type in the rejection message.

Summary Assessment

Verdict: Proceed with changes. The design is fundamentally sound but has issues that must be resolved before production deployment. The document processing pipeline, multi-tenant config, and OCR/classification tier strategy are correct. The three things I'd fix before this ever touches production Drive data:

  1. Persist pending state. SQLite file. $0 cost, ~40 LOC, eliminates the crash-data-loss problem that will otherwise define the user experience.
  2. Per-client service accounts. The config schema already supports it. It's an operational task to create separate keys, not a code change. Do it before the second client is onboarded.
  3. Restructure the state machine. Per-chat state with explicit enum transitions. Eliminates the race condition that causes documents to be misattributed.

Should you start the test service? Yes — the test bot (AFH_22, chat -1004391863263) is the right environment to shake out the engineering issues. But do not start the Edmonds Villa production bot until the pending state is persistent and the service account is scoped to one tenant. The first production document should be filed with crash-safe persistence in place.

What's the upgrade path? Items SP-2 (state persistence) and SP-4 (state machine) can be implemented together — the SQLite-backed per-chat state is a single refactor. SP-3 (service accounts) is operational, not code. EI-2 (test paths) is a 5-minute fix. The DeepSeek question (EI-1) is a design taste call — it works, it's just extraneous complexity. These are B-grade issues in a B+ design. Fix them and you're ready for production.

📋 Recommended Action Order

  1. Before first test run: SP-4 (state machine restructure), EI-2 (test paths) — these don't affect the live deployment, do them in-code first
  2. Before production (Edmonds Villa): SP-2 (state persistence), SP-3 (per-client service accounts), EI-3 (file guards), EI-4 (doc ID logging)
  3. First production month: OP-1 (temp file cleanup), QF-1 (Tier 3 feedback), QF-2 (extension whitelist)
  4. Post-launch polish: SP-1 (roster cache), EI-1 (DeepSeek vs Haiku decision)
