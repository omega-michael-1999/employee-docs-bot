Employee Docs Bot — Architecture Document

Date: 2026-06-24  |  Version: 1.0  |  Status: Deployed, pre-production

System Overview

The Employee Docs Bot is a document filing system for Washington State adult family homes. It receives employee documents via Telegram, extracts text via OCR, classifies by employee and WAC category using a hybrid rules+LLM approach, confirms classifications with the user, and files documents into Google Drive with proper naming and metadata.

The system is designed as a single Python process per Telegram bot token, sharing a multi-tenant configuration file. It runs on a single Linux server (hive) under systemd.

Classification Tier Logic

The classification engine is the most architecturally significant component. It balances speed (rules), accuracy (LLM), and human oversight (manual correction).

Tier 1: Rules

  • Employee matching: Word-boundary regex against the extracted text and filename for each employee name in the roster. The employee key is the folder name lowercased (e.g., "solethu qoyi" from "Solethu Qoyi").
  • Category matching: Substring keyword arrays per WAC category (e.g., "cpr", "bls", "first aid" → "04 - CPR & First Aid").
  • Confidence check: If the match is weak (single-word name hit, generic keyword), the result is flagged as ambiguous and escalated to Tier 2 rather than accepted.

Tier 2: LLM (DeepSeek)

  • Called when rules produce ambiguous or partial results.
  • Sends the extracted text (first 3000 chars) plus the candidate employee/category from Tier 1 as a hint.
  • DeepSeek returns JSON with employee, category, description.
  • Output is validated against the known employee roster and category list before acceptance.
  • Network timeout: 30s.

Tier 3: Manual Correction

  • Triggered when both Tier 1 and Tier 2 fail.
  • Two-step text prompt: (1) "What's the employee's full name?" with optional "🆕 New Employee" button, (2) "What type of document is this?" with example hints.
  • User's free-text description is mapped to a WAC category via keyword matching (match_category).
  • Final confirmation button before filing.

Each client gets its own Telegram bot token and systemd service instance. The config.json and Drive service account are shared. Client isolation is by (a) chat ID routing and (b) Drive folder permissions on the SA.

Observability

  • Logging: Python logging module at INFO level, output to journald via systemd. View with journalctl -u employee-docs-bot -f.
  • Failed classifications: Logged at WARNING level with the error and document info.
  • Drive API errors: Caught and reported to the user via Telegram message + logged.
  • Service health: systemd auto-restart on failure (RestartSec=10). No external monitoring configured.

Glossary

TermDefinition
ClientAn adult family home (AFH) using the bot. Each has its own Telegram chat, Google Drive folder, and config entry.
DocumentA single file sent to Telegram — photo, PDF, or DOCX. Processed independently.
New EmployeeA name detected in a document that doesn't match any folder in the Drive roster. Triggers automatic folder creation.
ProviderAFH owner/administrator (currently Sandra). Subject to additional WAC requirements including 54-hour administrator training.
WAC CategoryOne of 8 document categories defined by Washington Administrative Code chapters 388-76 and 388-112A.
rules matchClassification by keyword matching against employee roster and category keyword list. Fast but may be ambiguous.
LLM matchClassification by DeepSeek LLM. Slower but handles edge cases rules miss.
