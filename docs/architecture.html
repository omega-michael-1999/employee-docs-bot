System Overview

The Employee Docs Bot is a document filing system for Washington State adult family homes. It receives employee documents via Telegram, extracts text via OCR, classifies by employee and WAC category using a hybrid rules+LLM approach, confirms classifications with the AFH administrator, and files documents into Google Drive with proper naming and metadata.

Designed as a single Python process per Telegram bot token, sharing a multi-tenant configuration file. Runs on a single Linux server (hive) under systemd.

Classification Tier Logic

Tier 1: Rules

  • Word-boundary regex match against extracted text and filename for each employee name.
  • Substring keyword arrays per WAC category (e.g., "cpr" → "04 - CPR & First Aid").
  • Weak matches (single-word name, generic keyword) flagged as ambiguous and escalated.

Tier 2: LLM (DeepSeek)

  • Called when rules produce ambiguous results. Receives extracted text + rules result as hint.
  • DeepSeek returns JSON: {"employee", "category", "description"}.
  • Output validated against known roster and categories before acceptance. 30s network timeout.

Tier 3: Manual Correction

  • Two-step text prompt: (1) employee name with optional "🆕 New Employee" button, (2) document type description.
  • Free-text description mapped to WAC category via match_category keyword lookup.
  • Final confirmation button before filing.

Observability

  • Logging: Python logging at INFO level to journald. View: journalctl -u employee-docs-bot -f.
  • Failed classifications: Logged at WARNING with error and document metadata.
  • Drive errors: Caught and reported to the user via Telegram + logged.
  • Service health: systemd auto-restart (RestartSec=10). Heartbeat monitoring: bot pings HEARTBEAT_URL every 5 minutes via background asyncio task. Monitoring service (UptimeRobot, healthchecks.io, etc.) alerts on missed pings.
  • Heartbeat config: Set HEARTBEAT_URL in .env. Example: HEARTBEAT_URL=https://hc-ping.com/your-uuid. If unset, monitoring is disabled.
