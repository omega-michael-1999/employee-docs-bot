Employee Docs Bot — Product Requirements Document

Date: 2026-06-24  |  Status: Ready for implementation

Problem Statement

Sandra Namwase operates Edmonds Villa, an adult family home in Edmonds, WA. She is responsible for maintaining complete personnel files for ~16 employees against Washington State WAC 388-76 and 388-112A requirements. Currently, employee documents (CPR cards, TB test results, background checks, licenses, training certificates) arrive as photos and PDFs via Telegram. Sandra manually classifies each document by employee and category, renames it, and uploads it to the correct Google Drive folder. This takes 2–5 minutes per document, is error-prone, and creates a backlog when documents arrive in batches.

The same problem exists for new AFH clients onboarding. Each client needs the same document filing workflow with their own Telegram channel and Drive folder. The solution must be multi-tenant from the start.

Solution

A Telegram bot that receives employee documents (photos, PDFs, DOCX), automatically extracts text via OCR, classifies by employee name and WAC document category using a hybrid rules + LLM approach, confirms the classification with Sandra via inline buttons, and files the document into the correct Google Drive folder with proper naming. Sandra's interaction is minimal: send photo → tap 👍 or 👎 → done.

User Stories

  1. As Sandra, I want to send a photo of a CPR card to a Telegram channel and have it automatically filed to the correct employee's CPR folder in Drive, so that I don't have to manually rename and upload each document.
  2. As Sandra, when the bot is uncertain about the employee or document type, I want to see a confirmation prompt with inline buttons, so that I can quickly correct any misclassification before filing.
  3. As Sandra, when the bot cannot identify the employee at all, I want to type the employee's name and document type, so that new or unrecognized staff documents still get filed correctly.
  4. As Sandra, when a document is for a new employee not yet in the Drive roster, I want the bot to create their folder structure automatically, so that I don't have to set up folders manually.
  5. As Sandra, I want the bot to work with HEIC photos from my iPhone, so that I can send documents directly from my camera without converting formats.
  6. As Sandra, I want to send multiple documents in sequence and have each one processed independently, so that batch filing is natural.
  7. As Sandra, when the bot makes a classification error, I want to correct it via a simple text conversation (employee name → document type) rather than leaving the chat and editing Drive manually.
  8. As Sandra, I want the classified document to be filed as {Employee Name} - {Document Type}.{ext} with a companion -robot.txt containing the extracted text, so that the Drive is searchable and auditable.
  9. As an AFH owner onboarding a new client, I want to add a new client's Telegram chat ID and Drive folder to a config file, so that the same bot serves multiple AFHs without separate deployments.
  10. As an auditor, I want to run a compliance check against WAC 388-76 that knows which employees are providers (checked against categories 01–08 + facility docs) vs. caregivers (01–07 only), so that the audit respects regulatory differences.
  11. As Michael (operator), I want the bot to run as a systemd service that auto-restarts on failure and logs to journald, so that it stays up without manual intervention.
  12. As Michael (operator), I want the bot's classification to use rules for fast common cases and fall back to an LLM for ambiguous ones, so that accuracy is high without paying for an LLM call on every document.
  13. As Michael (operator), I want the bot to handle the test AFH_22 client and the production Edmonds Villa client as separate Telegram bot instances sharing the same config, so that testing doesn't interfere with live filing.
  14. As Michael (operator), I want to manage the deployment via /opt/employee-docs-bot/ as a git checkout owned by my user, so that I can browse, edit, and git pull without permission hassles.

Implementation Decisions

Architecture

  • Single bot binary, one systemd instance per Telegram token. The same bot.py and config.json are shared. Each instance reads its own .env with a different TELEGRAM_BOT_TOKEN. Chat ID routing in config.json distinguishes clients.
  • Multi-tenant from day one. config.json contains an array of client objects, each with chat_id, drive_root_id, provider list, and category keywords. No code changes needed to add a client.
  • Deployed at /opt/employee-docs-bot/ as a git checkout of omega-michael-1999/employee-docs-bot. Systemd service runs as michael user. Python venv at .venv/.
  • Shared Google service account. One SA key (employee-docs-bot@claude-ai-474917.iam.gserviceaccount.com) has Editor access to all client Drive folders. Folder-level permissions control isolation, not per-client SAs.

Document Processing Pipeline

  1. Receive — Telegram photo, PDF, or image file. HEIC supported via pillow-heif conversion to JPEG.
  2. Extract text — doc-extract.py: PyPDF2 for PDF text, Anthropic Haiku vision API for scanned PDFs and images, native for DOCX/TXT. 300s timeout for large docs.
  3. Classify — Three-tier hybrid approach:
    • Tier 1: Rules — word-boundary keyword matching against employee roster and category keywords.
    • Tier 2: LLM — if employee or category is ambiguous (partial name, weak keyword, multiple candidates), the rules result is treated as a hint and the full text is sent to DeepSeek for classification.
    • Tier 3: Manual — if LLM also fails, Sandra is prompted via inline buttons to enter the employee name and document type.
  4. Confirm — Inline keyboard [👍 Yes] [👎 No]. Correction flow: (1) type employee name, (2) type document description, (3) bot maps to WAC category via keyword matching, (4) final confirmation.
  5. File — Upload to DRIVE_ROOT/CAREGIVERS/{Employee}/{WAC Category}/{Name} - {Description}.{ext}. Companion -robot.txt with extracted text always uploaded alongside.
  6. Onboard new employee — If name not found in Drive roster, bot creates employee folder with all 8 WAC category subfolders and files the document.

Provider vs. Caregiver Model

  • No structural folder split. Both providers and caregivers use the same 8-category folder structure under CAREGIVERS/.
  • Config marks providers. "providers": ["Sandra Namwase"] in the client config.
  • Audit-level distinction. Compliance checks verify providers against categories 01–08 + facility-level docs (liability insurance, succession plan, emergency plan, fire drills). Caregivers are checked against 01–07 only.
  • Category 08 — Administrator Training (54-hour DSHS-approved course) is required for providers only, per WAC 388-112A-0800–0840.

Testing Decisions

Testing Seams

The highest-value testing seams, in priority order:

  1. End-to-end document flow — Send a real document via the test Telegram channel, verify classification, confirmation, and Drive filing. This is the primary seam and catches the most real-world issues.
  2. doc-extract.py standalone — Test against known file types (PDF text, scanned PDF, JPEG, HEIC, DOCX, PNG, TIFF) with known text content to verify extraction accuracy. Existing prior art: the script was forked from ai-os/scripts/doc-extract.py.
  3. Classification logic (classify_by_rules and classify_by_llm) — Unit tests with known employee rosters and document text to verify correct tier selection and fallback. This is where the ambiguity detection lives and is worth testing independently.
  4. Drive API integration — Test folder listing, creation, and file upload against a test Drive folder. This can be tested via the SA against the AFH_22 test folder.
  5. Confirmation flow state machine — The pending dict transitions (new doc → awaiting_employee → awaiting_category → confirm → file) should be tested for all branches including corrections, new employees, and cancel/expiry.

What Makes a Good Test

  • Tests external behavior, not implementation details. For doc-extract.py, test that text is extracted — not which internal method fired.
  • Test the classification tier selection: given known-good text, rules should match; given ambiguous text, LLM should be invoked.
  • Test real file types (HEIC, scanned PDF) not just convenient ones.
  • Drive integration tests use the real API against the test folder — no mocking the Drive service unless unavoidable.

Out of Scope

  • Persistent pending state — If the bot restarts mid-confirmation, the user re-sends the document. In-memory dict accepted for v1.
  • Batch document processing — Each document is processed independently. No "process all unread messages" command.
  • Audit report generation — The bot files documents; audit checks remain a separate process (the existing /employee-records-audit skill or manual review).
  • Dashboard or web UI — All interaction is via Telegram inline buttons and text.
  • User authentication — The bot trusts all messages in the configured channel. No per-user access control.
  • Provider folder under PROVIDER/ — Filing goes to CAREGIVERS/ for everyone. Provider distinction is audit-level only.

Further Notes

  • The ANTHROPIC_VISION_API_KEY is set and the pillow-heif library is installed in both venvs. No further setup needed for HEIC/OCR support.
  • Two Telegram bots exist: afh_22_employee_paperwork_bot (test) and edmonds_villa_paperwork_bot (production). Each needs its own systemd instance and .env file.
  • WAC 388-76-10198 (personnel records) is one of the most-cited violations by DSHS — this bot directly addresses that compliance risk by ensuring every document is filed to the correct category folder.
  • The AFH_22 test structure already has a CAREGIVERS/Test Employee, CNA/ folder with 7 category subfolders for testing.
