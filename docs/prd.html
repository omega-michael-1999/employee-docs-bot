Product Requirements

Employee Docs Bot — PRD

Automated document filing for AFH employee records

2026-06-24  ·  Status: Ready for implementation
Glossary PRD Architecture

Problem Statement

Adult family homes (AFHs) in Washington State must maintain complete personnel files for every employee against WAC 388-76 and 388-112A requirements. Employee documents (CPR cards, TB test results, background checks, training certificates) arrive as photos and PDFs via Telegram. The AFH administrator manually classifies each document by employee and category, renames it, and uploads it to the correct Google Drive folder — taking 2–5 minutes per document. This is error-prone and creates backlog when documents arrive in batches.

Omega Digital Services operates this bot as a multi-tenant service. Each AFH client gets their own Telegram channel, Drive folder, and bot token. New AFH clients are onboarded by the Omega Digital administrator — the bot's code and config are managed centrally.

Solution

A Telegram bot that receives employee documents, automatically extracts text via OCR, classifies by employee name and WAC document category using a hybrid rules + LLM approach, confirms the classification with the AFH administrator via inline buttons, and files to the correct Google Drive folder. The AFH administrator's interaction: send photo → tap 👍 or 👎 → done.

User Stories

  1. As an AFH administrator, I want to send a photo of a CPR card to a Telegram channel and have it automatically filed to the correct employee's CPR folder in Drive, so that I don't have to manually rename and upload each document.
  2. As an AFH administrator, when the bot is uncertain about the employee or document type, I want to see a confirmation prompt with inline buttons, so that I can quickly correct any misclassification before filing.
  3. As an AFH administrator, when the bot cannot identify the employee at all, I want to type the employee's name and document type, so that new or unrecognized staff documents still get filed correctly.
  4. As an AFH administrator, when a document is for a new employee not yet in the Drive roster, I want the bot to create their folder structure automatically, so that I don't have to set up folders manually.
  5. As an AFH administrator, I want the bot to work with HEIC photos from my iPhone, so that I can send documents directly from my camera without converting formats.
  6. As an AFH administrator, I want to send multiple documents in sequence and have each one processed independently, so that batch filing is natural.
  7. As an AFH administrator, when the bot makes a classification error, I want to correct it via a simple text conversation rather than editing Drive manually.
  8. As an AFH administrator, I want the classified document to be filed with a companion -robot.txt containing the extracted text, so that the Drive is searchable and auditable.
  9. As the Omega Digital administrator, I want to add a new AFH client's Telegram chat ID and Drive folder to a config file, so that the same bot serves multiple AFHs without separate deployments.
  10. As an AFH administrator, I want to run a compliance check against WAC 388-76 that knows which employees are providers (01–08 + facility docs) vs. caregivers (01–07 only), so that the audit respects regulatory differences.
  11. As the Omega Digital administrator, I want the bot to run as a systemd service that auto-restarts on failure and logs to journald, so that it stays up without manual intervention.
  12. As the Omega Digital administrator, I want classification to use rules for fast common cases and fall back to an LLM for ambiguous ones, so that accuracy is high without paying for an LLM call on every document.
  13. As the Omega Digital administrator, I want the test and production clients as separate Telegram bot instances sharing the same config, so that testing doesn't interfere with live filing.
  14. As the Omega Digital administrator, I want the deployment at /opt/employee-docs-bot/ as a git checkout owned by my user, so that I can browse, edit, and git pull without permission hassles.

Implementation Decisions

Architecture

  • Single bot binary, one systemd instance per token. The same bot.py and config.json are shared. Each instance reads its own .env with a different TELEGRAM_BOT_TOKEN. Chat ID routing in config.json distinguishes clients.
  • Multi-tenant from day one. config.json contains an array of client objects. No code changes needed to add a client. The Omega Digital administrator onboards new clients.
  • Shared Google service account. One SA key with Editor access to all client Drive folders. Folder-level permissions control isolation.

Document Processing Pipeline

  1. Receive — Telegram photo, PDF, or image file. HEIC supported via pillow-heif.
  2. Extract text — doc-extract.py: PyPDF2 for PDF text, Anthropic Haiku vision for scanned, native DOCX/TXT. 300s timeout.
  3. Classify — Three-tier: rules → LLM (if ambiguous) → manual correction.
  4. Confirm — Inline keyboard [👍 Yes] [👎 No]. Text correction flow if wrong.
  5. File — Upload to DRIVE_ROOT/CAREGIVERS/{Employee}/{Category}/{Name} - {Description}.{ext} + companion -robot.txt.
  6. Onboard new employee — Create employee folder with all 8 WAC subfolders if name not found.

Provider vs. Caregiver Model

  • No structural folder split. All filing under CAREGIVERS/.
  • Config marks providers in "providers": ["Sandra Namwase"].
  • Audit checks providers against 01–08 + facility docs; caregivers against 01–07.

Testing Decisions

Seams (highest to lowest):

  1. End-to-end: Send a real document via test Telegram channel, verify classification, confirmation, and Drive filing.
  2. doc-extract.py standalone: Test against PDF text, scanned PDF, JPEG, HEIC, DOCX, PNG, TIFF.
  3. Classification logic: Unit tests for classify_by_rules and classify_by_llm with known rosters and text.
  4. Drive API integration: Folder listing, creation, and upload against the AFH_22 test folder.
  5. Confirmation flow state machine: pending dict transitions — all branches including corrections, new employees, and expiry.

What makes a good test: Tests external behavior, not internals. Tests tier selection (known-good text → rules, ambiguous → LLM). Uses real file types (HEIC, scanned PDF). Drive tests use real API against test folder.

Out of Scope

  • Persistent pending state — in-memory dict accepted for v1; restart = re-send document
  • Batch document processing — each document processed independently
  • Audit report generation — bot files documents; audits remain a separate process
  • Dashboard or web UI — all interaction via Telegram inline buttons and text
  • User authentication — bot trusts all messages in the configured channel
  • Provider/PROVIDER folder routing — all filing goes to CAREGIVERS/

Further Notes

  • The ANTHROPIC_VISION_API_KEY is set and pillow-heif is installed in both venvs.
  • Two Telegram bots exist: afh_22_employee_paperwork_bot (test) and edmonds_villa_paperwork_bot (production). Each needs its own systemd instance.
  • WAC 388-76-10198 (personnel records) is one of the most-cited DSHS violations — this bot directly addresses that compliance risk.
  • AFH_22 test folder has CAREGIVERS/Test Employee, CNA/ with 7 category subfolders for testing.
